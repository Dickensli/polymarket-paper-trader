## Executive summary

The prior server changes are active: strategy-specific edge gates reject false or insufficient edge, eligible paper orders still fill, and a bounded Kalshi real BUY can be classified as a short-cover risk reduction. The remaining failure was agent-side eligibility drift. Three Kalshi paper strategies continued to treat historical report prose and unmet graduation metrics as permanent paper-trading halts, and some attempted to manufacture decision volume. This cycle makes reports output-only during trading bootstrap, marks graduation explicitly non-blocking for paper strategies, removes graduation calls from the two paper strategies that do not need notification, and strengthens all paper prompts against volume-building.

The real Kalshi route also submitted a depth-weighted average price such as `0.245098` as the FOK limit. That is neither the worst consumed ask required to cross the full simulated depth nor a valid cent tick. The route now retains the weighted average for risk/audit calculations, submits the worst consumed ask as the limit, and rounds Kalshi order prices conservatively to cent ticks.

Historical reports were not deleted. They remain immutable audit evidence but are no longer returned during an MCP trading bootstrap, and prompts forbid listing or reading them for trading decisions.

## Changes

### Output-only report memory

- MCP trading bootstrap continues to return authoritative registration, portfolio, positions, trades, orders, warnings, and run ID.
- When `start_run=true`, `recent_reports` is now empty and `report_memory_policy` is `output_only_not_loaded_for_trading`.
- Explicit non-trading audit requests can still list report filenames, and dedicated report tools remain available for human-directed audits.
- All paper prompts prohibit `list_reports`/`read_report` during a trading cycle and prohibit using report prose as current eligibility, risk, or market evidence.

### Paper graduation semantics

- Server graduation results now always include `blocksPaperTrading: false` and `paperTradingAllowed: true`.
- `commander` calls graduation only after execution decisions, for notification to the human.
- Kalshi `conservative_retro` and `high_freq_retro` no longer call graduation at all.
- Paper prompts state that graduation, Brier score, decision counts, policy history, and old reports never pause paper trading; only current market/risk validation may reject a current order.
- Strategies must not trade, submit, or document rejected decisions merely to build activity or graduation volume.

### Kalshi official FOK pricing

- Risk and proposal validation continue to use the depth-weighted simulated average price.
- Official Kalshi BUY submission uses the highest ask consumed by the full shadow fill as its FOK limit.
- Kalshi BUY prices round upward and SELL prices downward to legal cent ticks before YES-side conversion.
- Audit requests record both `price` (weighted average) and `submission_limit_price`.
- `commander_real` now explicitly treats public live books and the authenticated Demo execution book as separate liquidity environments.

## Rejected changes

- Deleting historical reports was rejected because deletion removes evidence without fixing the ingestion path.
- Forcing paper trades to prove activity was rejected because it corrupts the evaluation sample and increases risk.
- Loosening commander or high-frequency risk limits was rejected because the sample is too short and current server gates are working.
- Treating Demo FOK insufficient volume as a venue outage was rejected because live public depth does not exist in the Demo execution book.

## Expected metrics and rollback triggers

- Expected: no new paper report cites unmet graduation fields or prior report conclusions as the reason for a permanent halt.
- Expected: no paper strategy calls report read/list tools during normal bootstrap or creates decisions solely for volume.
- Expected: eligible current opportunities continue to reach server validation; server rejections remain attributable to current proposal, liquidity, or risk fields.
- Expected: no Kalshi official order fails with `invalid_price` because an average fill was submitted off tick.
- Roll back report isolation only if a required current position/trade field is proven absent from authoritative APIs; never restore narrative report memory as the source of truth.
- Roll back tick handling if Kalshi rejects a valid cent-tick request or the conservative rounding direction makes a risk-reducing order less safe.
